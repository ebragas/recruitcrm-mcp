So the bug is specifically about including `about_company` in the create payload. Likely fix: drop `about_company` from create payloads in `client.create_company` callers, or document the gotcha.

- [ ] **Read-after-write race in update round-trip tests.** `TestUpdateContactRoundTrip::test_partial_update_preserves_other_fields` (and likely TestUpdateCandidateRoundTrip / TestUpdateMeetingRoundTrip — same pattern) fails intermittently because `GET /contacts/{slug}` immediately after `POST /contacts/{slug}` returns stale data. Direct API query ~30s later shows the update did persist (test #4's slug `17773003273710031585RYL` reads back all 4 seeded fields). Fixes: (a) add a small `await asyncio.sleep(1)` between update and read, (b) retry the read with backoff until expected value appears, or (c) document this as a known limitation and skip the baseline assertion (only check the final state). Option (b) is most robust.

- [ ] **`make integration-sweep` should require per-entity confirmation before deleting.** Current behavior: lists matches, then deletes them all in one shot with no chance to abort. Scary on a shared production tenant — a single false-positive prefix match would silently destroy real data. Add a `--dry-run` flag (default for human runs) that lists what would be deleted and exits, plus an interactive `--confirm` mode that prompts `delete X? [y/N]` per entity. Only the autouse session-end fixture should auto-delete (since it's already gated to test entities the same session created/named). Bonus: add a max-delete safety cap (e.g. abort if >50 matches).
